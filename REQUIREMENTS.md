# 求人マッチングサイト 要件定義書

## 1. プロジェクト概要
- **種別**: 求人マッチングサイト(求人サイトをベースにしたマッチングサービス)
- **対象職種**: エンジニア職に限定
- **目的**: 学習・ポートフォリオ
- **プラットフォーム**: Webアプリ

## 2. ユーザーロール
- **求職者(Job Seeker)**: 求人を検索し、応募する側
- **企業(Company)**: 求人を掲載し、応募者を選考する側

## 3. マッチングフロー
検索・応募型。「いいね」は求職者側からのみ行え、それ自体が応募の意思表示となる(企業からのスカウトはなし)。

1. 求職者が求人を検索・閲覧
2. 気になる求人に「いいね」(=応募)
3. 企業が7日以内に「気になる」を返す
   - 返した場合: マッチ成立し、企業⇔求職者でメッセージ開始
   - 7日以内に返さなかった場合: 自動的にマッチ不成立(それ以降の進展はなし)
4. マッチ成立後、企業が選考ステータスを更新しながら求職者とメッセージのやり取り
5. 面接日程調整(候補日提示 → 確定)
6. 選考結果連絡

## 4. 機能要件

### 4.1 共通
- 会員登録・ログイン・ログアウト(Laravel Sanctum, SPA認証)
- パスワードリセット
- プロフィール編集

### 4.2 求職者向け機能
- プロフィール作成・編集(氏名・経歴・スキル・希望条件 等)
- 求人検索・絞り込み(キーワード・職種・勤務地・給与 等)
- 求人詳細閲覧
- 求人への「いいね」(=応募)
- 応募状況(選考ステータス・マッチ成立可否)確認
- 企業とのメッセージ
- 面接日程調整(候補日確認・確定)
- 通知(応募結果・新着メッセージ等。アプリ内通知 + メール通知)

### 4.3 企業向け機能
- 企業プロフィール作成・編集
- 求人の投稿・編集・削除・公開/非公開切り替え
- 応募者(いいねしてきた求職者)一覧・プロフィール閲覧
- 応募者への「気になる」でマッチ成立(7日以内に反応しないと自動不成立)
- 選考ステータス管理(例: 書類選考中 / 面接調整中 / 内定 / 不採用)
- 求職者とのメッセージ
- 面接日程調整(候補日提示・確定)
- 通知(応募・選考結果・新着メッセージ等。アプリ内通知 + メール通知)

### 4.4 マネタイズ(企業向け課金)
求人1件ごとの個別サブスクリプション制。学習目的のため決済はStripeのテストモードで実装する。

- 求人を掲載すると、掲載日から**14日間は無料**
- 無料期間中にStripeで支払い方法(カード)を登録
- 無料期間終了時点で有効な支払い方法があれば、その求人単位で**月額1,000円**のサブスクリプション課金を開始し、以後は自動更新(毎月課金)
- 無料期間終了時点で支払い方法が未登録、または決済に失敗した場合、その求人は**自動的に非公開**になる(データは保持され、後から支払い登録すれば再公開可能)
- 企業は求人ごとの課金ステータス(無料期間中 / 課金中 / 未払いで非公開)と請求履歴を確認できる

### 4.5 スコープ外(今回のMVPには含めない)
- 企業から求職者への「スカウト」機能
- 有料プッシュ通知・SMS通知(FCM等)
- レコメンド(自動推薦)アルゴリズム
- 管理者ダッシュボード

## 5. 非機能要件
- レスポンシブ対応(PCブラウザ想定、スマホも考慮)
- ローカル開発からDockerを使用
- Railwayへデプロイ
- セキュリティ: CSRF対策(Sanctum標準)、XSS対策、入力バリデーション

| 領域 | 技術 |

| Frontend | Next.js |

## 7. 主要データモデル(ドラフト)
- `users`(求職者)
- `companies`(企業。1社1アカウントとしてログイン主体を兼ねる)
- `job_postings`(求人。公開/非公開ステータスを含む)
- `applications`(求職者の「いいね」= 応募。マッチ成立可否・選考ステータスを含む)
- `messages`(応募単位でのメッセージスレッド)
- `interview_schedules`(面接日程調整。候補日〜確定日)
- `notifications`
- `job_posting_subscriptions`(求人ごとのStripeサブスクリプション。trial_ends_at, stripe_subscription_id, status 等)
- `payments`(請求履歴。Stripe Webhookで同期)
